# НПТ Навигатор

НПТ Навигатор — независимый демонстрационный MVP цифрового помощника по подготовке научно-популярного маршрута.

Проект показывает, как сложные методические материалы инициативы «Научно-популярный туризм» и структуру Приложения 2 можно превратить в понятный интерфейс: с объяснениями, примерами, самопроверкой и рабочей зоной подготовки черновика маршрута.

> Важно: это не официальный сайт Десятилетия науки и технологий, не официальный сервис инициативы НПТ и не замена процедуре верификации маршрутов.

## Цель проекта

Главная задача проекта — не просто помочь пользователю заполнить поля, а объяснить:

- что хотят увидеть эксперты;
- как понять требования Приложения 2;
- как отличить сильное описание маршрута от слабого;
- какие типовые ошибки могут помешать верификации;
- как подготовить более качественный черновик описания маршрута.

Сквозной принцип интерфейса:

> «Что от меня хотят эксперты и как понять, что я сделал это хорошо?»

## Для кого проект

Основные пользователи MVP:

- университеты и научные организации;
- музеи и научно-популярные площадки;
- туроператоры;
- региональные органы власти;
- авторы и разработчики маршрутов.

Проект рассчитан на пользователя, у которого уже есть идея маршрута, но нет уверенности, что маршрут соответствует логике научно-популярного туризма и требованиям Приложения 2.

## Образовательные карточки

Образовательная карточка — ключевая единица «машины понимания».

Каждая карточка должна помогать пользователю:

- понять смысл раздела;
- увидеть ожидания экспертов;
- сравнить хороший и слабый пример;
- избежать типовых ошибок;
- пройти самопроверку;
- перейти к заполнению рабочей зоны.

Сейчас реализованы карточки:

- Общая информация
- Цели и задачи
- Научная составляющая
- Программа маршрута
- Образовательный компонент
- Научно-популярные объекты
- Интерактивная составляющая
- Техническое обеспечение

## Структура проекта

```text
.
├── index.html
├── assets
│   ├── css
│   │   └── styles.css
│   └── js
│       └── app.js
└── info
    ├── *.md
    ├── *.docx
    └── *.xlsx
```

### Основные файлы

- `index.html` — основной HTML страницы и образовательные лендинговые секции.
- `assets/css/styles.css` — вся визуальная система, адаптив и стили компонентов.
- `assets/js/app.js` — данные, состояние формы, рендер карточек, sidebar, timeline, диагностика и экспорт.
- `info/` — проектные материалы, документы, архитектура, контент-модель и исходные методические файлы.

## Как запустить

Проект не требует сборки и зависимостей.

Самый простой способ:

```bash
open index.html
```

Или открыть `index.html` напрямую в браузере.

Если нужен локальный сервер:

```bash
python3 -m http.server 8000
```

Затем открыть:

```text
http://localhost:8000
```

## Как устроена логика

### Состояние

Основное состояние формы хранится в `state` внутри `assets/js/app.js`.

Текущий раздел задаётся переменной `currentStep`.

От `currentStep` синхронно зависят:

- активный пункт sidebar;
- активная точка timeline;
- заполненная синяя линия timeline;
- подпись текущего блока;
- содержимое рабочей карточки.

### Рабочая зона

Каждый раздел рендерится через объект `renderers` в `assets/js/app.js`.

Например:

- `general` — «Общая информация»;
- `goals` — «Цели и задачи»;
- `science` — «Научная составляющая»;
- `program` — «Программа маршрута».

### Экспорт

Итоговый текст собирается функцией `buildExportText()` в `assets/js/app.js`.

Важно: итоговый документ следует воспринимать как черновик описания маршрута на основе Приложения 2, а не как официальную заявку.

## Принципы дальнейшей разработки

1. Не превращать проект в обычную форму заполнения.
2. Не превращать проект в длинную методичку.
3. Сохранять фокус на понимании требований.
4. Каждый новый раздел должен отвечать на вопрос: что хочет увидеть эксперт?
5. Форма и экспорт — вспомогательные инструменты, а не главная ценность продукта.
6. Использовать только официальные названия разделов Приложения 2.
7. Не ломать существующую навигацию, timeline, sidebar и экспорт.

## Рекомендуемый шаблон новой карточки

Для следующих разделов использовать уже сложившуюся структуру:

1. Что хотят увидеть эксперты
2. Простое объяснение
3. Хороший пример
4. Слабый пример
5. Типовые ошибки
6. Самопроверка
7. Связь с рабочей зоной

Карточка должна появляться в начале соответствующего раздела рабочей зоны, перед полями заполнения.

## Проверка после изменений

Минимальная проверка:

```bash
node --check assets/js/app.js
```

Также нужно вручную или через headless browser проверить:

- сайт открывается без ошибок;
- active-состояния timeline и sidebar совпадают;
- поля ввода сохраняют данные;
- чипы и чек-листы работают;
- итоговый черновик собирается;
- мобильная версия не даёт горизонтального overflow.

## Статус проекта

Проект находится в стадии демонстрационного MVP.

Текущий фокус разработки:

- завершить образовательные карточки для всех 13 разделов;
- усилить самопроверку;
- связать карточки с итоговым чек-листом готовности;
- улучшить итоговый черновик описания маршрута;
- подготовить цельную демонстрационную версию.

## Дисклеймер

НПТ Навигатор — независимый концепт-прототип. Он не является официальным сервисом, не отправляет данные на сервер и не заменяет официальные процедуры согласования, подачи и верификации научно-популярных маршрутов.
