# Грантмастер

Грантмастер — автономный интерактивный помощник по проектированию инициатив, социальных, культурных, образовательных, молодёжных, технологических, инфраструктурных и грантовых проектов.

Проект помогает пользователю не “написать красивую заявку”, а собрать проектную логику: понять проблему, аудиторию, решение, цель, задачи, реализацию, результаты, ресурсы, поддержку и устойчивость проекта.

> Важно: Грантмастер — не AI-генератор заявок, не CRM, не каталог грантов и не платформа подачи заявок. Это локальный экспертно-методический конструктор проектного мышления.

## Главный принцип

Сначала человек понимает проект.  
Потом готовит материалы для заявки.

Интерфейс построен вокруг принципа:

> “Сначала объясни — потом попроси заполнить”.

Пользователь должен как можно меньше писать с нуля и как можно больше выбирать, уточнять, структурировать и проверять связи между частями проекта.

## Что делает продукт

Грантмастер помогает:

- определить стартовую зрелость проекта;
- собрать краткий паспорт проекта;
- сформулировать проблему и подтвердить её актуальность;
- описать целевую аудиторию и её потребности;
- связать проблему, решение, цель, задачи и мероприятия;
- продумать результаты и способы их проверки;
- показать команду, партнёров, ресурсы и поддержку;
- описать устойчивость проекта после финансирования;
- собрать дополнительные подтверждающие материалы;
- сформировать итоговый черновик паспорта проекта.

## Что не делает продукт

Грантмастер не:

- отправляет заявки в конкурсы;
- хранит данные на сервере;
- использует внешние API;
- авторизует пользователей;
- подбирает конкурсы;
- генерирует заявку вместо автора;
- заменяет требования конкретного грантодателя.

Все данные остаются локально в браузере пользователя.

## Текущая структура конструктора

Рабочая зона конструктора состоит из 15 разделов:

1. Первичная диагностика
2. Паспорт проекта
3. Проблема
4. Целевая аудитория
5. Концепция проекта
6. Цель и задачи
7. Реализация
8. Результаты
9. Команда
10. Партнёры
11. Продвижение
12. Ресурсы
13. Поддержка
14. Устойчивость
15. Дополнительные материалы

После рабочих разделов расположена отдельная финальная зона конструктора с индикатором заполненности и переходом к формированию итогового паспорта проекта.

## Образовательные карточки

Образовательные карточки — отдельный методический слой конструктора. Они появляются перед рабочими полями раздела и помогают пользователю понять смысл блока до заполнения.

Утверждённая структура карточки:

1. Краткое объяснение, зачем нужен раздел.
2. “Что хотят увидеть эксперты”.
3. Хороший пример.
4. Слабый пример.
5. Типичные ошибки.
6. Самопроверка.
7. “Как работать дальше”.

На текущем этапе реализованы карточки:

- Первичная диагностика
- Паспорт проекта
- Проблема

Карточки не должны превращаться в длинную методичку. Их задача — быстро обучить логике раздела и подготовить пользователя к заполнению полей.

## Итоговый паспорт проекта

Итоговый паспорт проекта — это отдельный экран с черновиком структурированного описания проекта.

Он позволяет:

- посмотреть собранные данные по всем разделам;
- увидеть незаполненные поля;
- скопировать текст;
- скачать `.txt`-версию;
- использовать черновик для дальнейшей адаптации под конкретный конкурс, партнёра или внутреннее обсуждение.

Итоговый паспорт не является официальной заявкой и не заменяет конкурсную форму.

## Техническая база

Проект полностью статический:

- HTML
- CSS
- JavaScript
- без сборки;
- без сервера;
- без базы данных;
- без авторизации;
- без внешних API.

Основные файлы:

```text
.
├── index.html
├── assets
│   ├── css
│   │   └── styles.css
│   └── js
│       └── app.js
├── info
│   ├── ПАСПОРТ ПРОЕКТА по состоянию на 11.07.2026.txt
│   ├── методические материалы
│   └── Skrin
└── README.md
```

### `index.html`

Основная разметка страницы: главная, диагностический блок, карта логики проекта, рабочая зона конструктора и финальные области.

### `assets/css/styles.css`

Визуальная система проекта: графитово-оранжевая палитра, карточки, адаптив, навигация, образовательные блоки, рабочие формы и итоговый паспорт.

### `assets/js/app.js`

Основная логика приложения:

- структура разделов;
- состояние `state`;
- рендер рабочей зоны;
- чипы, чекбоксы, dropdown, динамические строки;
- локальное сохранение данных;
- навигация по разделам;
- расчёт заполненности;
- образовательные карточки;
- формирование итогового паспорта;
- копирование и скачивание `.txt`.

### `info/`

Папка с проектными и методическими материалами: паспорт проекта, концепция, материалы СИНВ, ПФКИ, Росмолодёжи, АСИ, чек-листы, примеры и визуальные референсы.

При разработке формулировок, справочников, подсказок и образовательных карточек нужно опираться на `info/`, а не придумывать профессиональную логику с нуля.

## Как запустить

Проект не требует установки зависимостей.

Самый простой способ — открыть файл:

```text
index.html
```

Если нужен локальный сервер:

```bash
python -m http.server 8000
```

Затем открыть:

```text
http://localhost:8000
```

## Проверка после изменений

Минимальная обязательная проверка для изменений в JavaScript:

```bash
node --check assets/js/app.js
```

Для интерфейсных изменений дополнительно желательно выполнить browser-check в Chrome:

- desktop;
- mobile;
- отсутствие ошибок;
- отсутствие очевидных проблем адаптивности.

Текущий рабочий порядок: делать не более 2–3 попыток автоматического browser-check. Если Chrome в среде не создаёт корректные скриншоты или требует долгой отладки, проверку не нужно превращать в отдельную задачу — достаточно честно указать это в отчёте.

## Принципы разработки

1. Не переписывать проект с нуля.
2. Не ломать текущие механики навигации, сохранения, рендера и экспорта.
3. Менять только то, что относится к текущей задаче.
4. Не трогать соседние разделы без необходимости.
5. Сохранять 15 рабочих разделов конструктора.
6. Не путать рабочие разделы и образовательные карточки.
7. Сохранять автономность: без сервера, базы данных, авторизации, внешних API и AI-функций.
8. Использовать существующие helper-функции и паттерны интерфейса.
9. Сохранять графитово-оранжевую стилистику, мягкие карточки, тонкие контуры и аккуратные скругления.
10. При проектировании содержания опираться на материалы `info/`.

## Статус проекта

Проект находится в стадии разработки MVP.

Уже выполнена трансформация исходного HTML/CSS/JavaScript-прототипа в автономный конструктор проектной логики и грантовых заявок.

Текущий фокус:

- продолжить образовательные карточки по разделам 4–15;
- сохранить компактность карточек;
- усиливать интерактивность разделов без перегруза;
- развивать итоговый паспорт проекта;
- поддерживать единый визуальный и методический язык продукта.

## Дисклеймер

Грантмастер — концепт-прототип и рабочий MVP для проектирования логики инициатив и подготовки материалов. Он не является официальным сервисом какого-либо грантодателя и не заменяет правила, положения и формы конкретных конкурсов.
